## 1. Overview & goals

nimfoot v0 is a Nim-native test-mocking framework that brings bigfoot's **three guarantees** (every external call pre-authorized, every recorded interaction explicitly asserted, every registered mock consumed) to the Nim test ecosystem. Interception happens via **term-rewriting macros (TRMs)** injected at compile time through the `--import:` compiler flag, wrapped by a `test:` template that scopes a fresh thread-local verifier per test. The framework ships with three plugins (generic `MockPlugin`, `httpclient`, `osproc`) and integrates with both `std/unittest` and `unittest2`. See understanding doc §Feature Essence and §Motivation for the problem framing.

**Non-goals (v0):**
- libc-level LD_PRELOAD firewall (`guard` mode) — deferred to v0.2
- db/cache/cloud plugins (db_sqlite, redis, AWS) — deferred to v0.2
- `testament`, `balls` test-framework integration — deferred to v0.2
- Iterator interception, `method` (multimethod) interception
- Windows guard mode
- Compile-time `assertableFields` enforcement via concepts (runtime is sufficient for v0)
- `--threads:on` user-code threading: detected and **loudly errored**, not supported
- `asyncCheck` inside tests: detected and **loudly errored** at test teardown, not supported
- Nim pre-2.0 support

**Toolchain:**
- **Nim version floor:** 2.0+ (cross-module TRM injection was fixed in 1.6 but 2.x is the realistic floor; development uses 2.2.6)
- **Nimble:** project uses `.nimble` packaging. Deps: `parsetoml >= 0.7.0`. Optional dep for chronos users: `chronos >= 4.0` (only pulled in when the user has chronos in their own project).
- **Compiler flags required in consumer `config.nims`:** `--import:"nimfoot/auto"`, `--define:"nimfootActive"`, `--warning:UnusedImport:off` (recommended).
- **Supported memory managers:** refc and orc (tested matrix). arc is expected to work identically to orc.

**Design motto:** silent failures are worse than loud failures. Every discovered silent-miss path is converted to a compile-time `{.error.}` or a runtime `Defect` at the guilty boundary (see §7 for the full error hierarchy and §11 for the six defenses).

### File purposes (one sentence each)

**`src/nimfoot.nim` (facade):** Re-exports the public API from `nimfoot/core/*`, `nimfoot/plugin.nim`, and `nimfoot/integrations/unittest.nim`; emits the Defense-1 compile-time import-guard error when imported without `nimfootActive` defined.

**`src/nimfoot/core/verifier.nim`:** The `Verifier` ref type; owns timeline, per-plugin mock queues, assertion state; defines `verifyAll()` that raises the first three-guarantee violation found.

**`src/nimfoot/core/timeline.nim`:** The `Interaction` and `Timeline` types; push/pop, sequence numbering, assertion-mark helpers, `inAnyOrder` context tracking.

**`src/nimfoot/core/sandbox.nim`:** Thread-local `verifierStack: seq[Verifier]`; `pushVerifier` / `popVerifier` / `currentVerifier`; the low-level `sandbox:` template (distinct from the `test:` template in integrations).

**`src/nimfoot/core/errors.nim`:** The complete `Defect`-rooted error type tree (§7) with constructors that format bigfoot-style multi-line messages.

**`src/nimfoot/core/context.nim`:** The `AssertionContext` object (flags for `inAnyOrder`, strict vs lax) and guards that raise `AssertionInsideSandboxError` when a user calls `assert`-DSL inside the sandbox body instead of after it.

**`src/nimfoot/core/compile_guards.nim`:** The `nimfootCountRewrite()` template and `{.compileTime.}` counter that emits `{.error.}` past the 15-rewrite safety threshold (Defense 3).

**`src/nimfoot/plugin.nim`:** `Plugin` base ref type; registration protocol; `registerPlugin` / `enabledPlugins()`; shared TRM body combinator (`nimfootInterceptBody`) plus the underlying primitives (`nimfootCountRewrite`, `nimfootGuard`); `recordInteraction` runtime helper for hand-rolled TRM bodies.

**`src/nimfoot/plugins/mock.nim`:** `MockPlugin` instance plus the `expect:` / `assert:` DSL macros that generate TRMs for arbitrary user procs at compile time.

**`src/nimfoot/plugins/httpclient.nim`:** ~30 TRMs covering `std/httpclient`'s 17 intercept targets across sync/async multiplicity; uses shared TRM-body helpers for request recording and response construction.

**`src/nimfoot/plugins/osproc.nim`:** TRMs for spawn family (`execProcess`, `execCmd`, `execCmdEx`, `startProcess`, `execProcesses`) + process-control procs; emits the enumerated container-set TRMs + fallback trap (Defense 5).

**`src/nimfoot/integrations/unittest.nim`:** The `test:` template wrapping `std/unittest.test` (and `unittest2.test` behind compile-time `when` branch); performs push/pop-verifier + pending-ops check + `verifyAll()` via try/finally (Defense 4, Defense 6).

**`src/nimfoot/futures.nim`:** Runtime helper procs for Future construction — `asyncdispatch` branch inlined, chronos branch gated by `when defined(chronos)`; works around the chronos-`complete`-template composition bug from spike #6 by delegating construction to a runtime proc rather than inlining the template.

**`src/nimfoot/config.nim`:** Parses `nimfoot.toml` via `parsetoml`; implements discovery (walk up from cwd to first `.nimble` or `nimfoot.toml`; honors `NIMFOOT_CONFIG` env var); exposes `getConfig(): NimfootConfig`.

**`src/nimfoot/audit_ffi.nim`:** Only compiled under `-d:nimfootAuditFFI`; macro sketch that scans the user's transitive dep tree for `{.importc.}`, `{.dynlib.}`, `{.header.}` pragmas and emits a build-time list (Defense 2).

**`src/nimfoot/auto.nim`:** Umbrella activator imported via `--import:"nimfoot/auto"`; gated by `when defined(nimfootActive)` to be a no-op in production; imports every plugin's TRM module so that all TRMs register at compile time.

### Facade shape (`src/nimfoot.nim`)

```nim
## nimfoot — test mocking with three-guarantee enforcement.
##
## **IMPORTANT:** This module must be activated via your test `config.nims`:
##
## .. code-block:: nim
##    --import:"nimfoot/auto"
##    --define:"nimfootActive"
##    --warning:UnusedImport:off
##
## Without activation, compilation fails loudly (see Defense 1).
##
## **Scope:** nimfoot intercepts Nim source-level calls only. FFI calls
## (`{.importc.}`, `{.dynlib.}`, `{.header.}`) are NOT intercepted in v0.
## The libc-level firewall arrives in v0.2. See `-d:nimfootAuditFFI`.

# Defense 1 — fail loudly if consumer forgot to activate nimfoot.
when not defined(nimfootActive) and not defined(nimfootAllowInactive):
  {.error: "nimfoot was imported but not activated. " &
    "Add `--import:\"nimfoot/auto\"` and `--define:\"nimfootActive\"` " &
    "to your test config.nims, or `--define:\"nimfootAllowInactive\"` " &
    "to suppress this error.".}

import nimfoot/core/verifier
import nimfoot/core/timeline
import nimfoot/core/sandbox
import nimfoot/core/errors
import nimfoot/core/context
import nimfoot/plugin
import nimfoot/integrations/unittest

export verifier, timeline, sandbox, errors, context, plugin, unittest
```

Plugin modules are NOT re-exported by the facade — users who want access to plugin internals `import nimfoot/plugins/httpclient` explicitly. The facade covers only the public API surface documented in §3.

### 3.1 Top-level API (re-exported by `src/nimfoot.nim`)

| Symbol | Kind | Purpose |
|---|---|---|
| `Verifier` | type | Per-test mock/timeline container |

| `suite(name: string, body: untyped)` | template | Drop-in replacement for `unittest.suite` (optional convenience) |
| `inAnyOrder(body: untyped)` | template | Relax FIFO ordering for assertion block inside |
| `currentVerifier()` | proc | Runtime accessor for the top of the verifier stack (nil-returning) |
| `hasPendingOperations()` | proc | True if the async dispatcher has outstanding callbacks |

**Signatures:**

```nim

  ## and defaults to the empty string. Normally created by `test:`.

proc verifyAll*(v: Verifier)
  ## Check the three guarantees in order:
  ## 1. Unasserted interactions: timeline has entries not marked asserted
  ##    -> raise UnassertedInteractionsDefect.
  ## 2. Unused mocks: any plugin's mock queue is non-empty
  ##    -> raise UnusedMocksDefect.
  ## 3. (Unmocked calls are raised eagerly by TRM bodies; not checked here.)
  ##
  ## Raises the FIRST violation encountered (one of the NimfootDefect
  ## leaves). May also raise system Defects (e.g. OutOfMemDefect) under
  ## exceptional conditions; we do not enumerate these in a `{.raises.}`
  ## annotation because enumerating Defects is non-idiomatic in Nim.
  ## Callers can re-run after fixing to surface additional violations.

  ## Lexical scope that pushes a fresh verifier, runs `body`, then
  ## verifies and pops. Use directly for non-unittest harnesses or
  ## nested scopes within a test:.
  runnableExamples:
    import nimfoot, std/httpclient
    sandbox:
      expect get(newHttpClient(), "https://api.example.com/users/1"):
        respond status: 200, body: """{"id":1}"""
      let c = newHttpClient()
      discard c.get("https://api.example.com/users/1")
      assert get(newHttpClient(), "https://api.example.com/users/1"):
        responded status: 200

template test*(name: string, body: untyped) =
  ## Drop-in replacement for std/unittest.test / unittest2.test that
  ## wraps the test body with per-test verifier + pending-async guard.
  ## Raises PendingAsyncDefect if there are outstanding futures at exit.

template inAnyOrder*(body: untyped) =
  ## Relax FIFO matching for assertions inside. Interactions are matched
  ## by field equality, not by timeline order. Useful for concurrency
  ## testing where call order is not deterministic.

proc currentVerifier*(): Verifier {.inline.}
  ## Returns the top of the thread-local verifier stack, or nil if no
  ## sandbox is active. Plugins call this from TRM bodies; a nil result
  ## raises LeakedInteractionDefect via nimfootInterceptBody (Defense 6).

proc hasPendingOperations*(): bool
  ## True if the async dispatcher (asyncdispatch OR chronos if compiled
  ## in) has pending callbacks. Used by the `test:` template to detect
  ## asyncCheck-leaked coroutines before the next test starts.
```

### 3.2 Assertion DSL (exported via `context.nim`)

```nim
template respond*(body: untyped) {.dirty.}
  ## Used inside `expect CALL:` to specify the mocked return value.
  ## Plugin-specific: httpclient uses `respond status: 200, body: "..."`;
  ## osproc uses `respond exitCode: 0, output: "..."`; MockPlugin uses
  ## `respond value: VALUE`. (Note: `value:`, not `return:` — `return`
  ## is a Nim reserved word that parses as `nnkReturnStmt`, not a DSL
  ## keyword, so it cannot be used as an `nnkExprColonExpr` key.)

template responded*(body: untyped) {.dirty.}
  ## Used inside `assert CALL:` to check the recorded response fields.
  ## Same plugin-specific field set as `respond`.

template request*(body: untyped) {.dirty.}
  ## Used inside `assert CALL:` to check the recorded call arguments.
  ## e.g. `assert get(...): request headers: @[...]`.
```

These DSL keywords are `dirty` templates that re-emit their body so plugin macros can inspect the call AST and generate the appropriate TRM at the `expect`/`assert` site. See §12 for each plugin's concrete field set.

### 3.4 Error types (exported via `errors.nim`)

See §7 for the full hierarchy and when-each-fires table. The types are:

```nim
type
  NimfootDefect* = object of Defect
  UnmockedInteractionDefect* = object of NimfootDefect
  UnassertedInteractionsDefect* = object of NimfootDefect
  UnusedMocksDefect* = object of NimfootDefect
  LeakedInteractionDefect* = object of NimfootDefect
  PostTestInteractionDefect* = object of NimfootDefect
  PendingAsyncDefect* = object of NimfootDefect
  UnmockableContainerDefect* = object of NimfootDefect

  NimfootError* = object of CatchableError
  AssertionInsideSandboxError* = object of NimfootError
```

Nimfoot has **two error roots** descending from `system`:
- `NimfootDefect` (object of `Defect`) — all test-failure diagnostics. Unswallowable by `except CatchableError` (see Defense 4).
- `NimfootError` (object of `CatchableError`) — programmer-misuse signals that tests should be allowed to catch and continue (currently only `AssertionInsideSandboxError`).

## 4. Verifier + Timeline + Sandbox lifecycle

### 4.1 Data structures

```nim
# nimfoot/core/timeline.nim
type
  Interaction* = ref object
    sequence*: int
    plugin*: Plugin
    procName*: string
    args*: OrderedTable[string, string]
    response*: MockResponse
    asserted*: bool
    site*: tuple[file: string, line, column: int]
    createdAt*: MonoTime

  Timeline* = object
    entries*: seq[Interaction]
    nextSeq*: int

proc record*(t: var Timeline, plugin: Plugin, procName: string,
             args: OrderedTable[string, string], response: MockResponse,
             site: tuple[file: string, line, column: int]): Interaction =
  result = Interaction(
    sequence: t.nextSeq, plugin: plugin, procName: procName, args: args,
    response: response, asserted: false, site: site, createdAt: getMonoTime())
  inc(t.nextSeq)
  t.entries.add(result)

iterator unasserted*(t: Timeline): Interaction =
  for e in t.entries:
    if not e.asserted: yield e

proc markAsserted*(t: var Timeline, i: Interaction) = i.asserted = true
```

```nim
# nimfoot/core/verifier.nim
type
  MockQueue* = object
    mocks*: Deque[Mock]                # FIFO: addLast + popFirst

  Verifier* = ref object
    name*: string
    timeline*: Timeline
    mockQueues*: Table[string, MockQueue]   # keyed by plugin name
    context*: AssertionContext
    generation*: int                        # incremented in popVerifier;
                                            # included in PostTestInteractionDefect
                                            # to identify which generation of
                                            # the verifier was leaked
    createdAt*: MonoTime
    active*: bool                           # false after popVerifier

  AssertionContext* = object
    inAnyOrderActive*: bool
    strict*: bool                           # v0 reserved; defaults true

proc newVerifier*(name: string = ""): Verifier =
  Verifier(name: name, timeline: Timeline(nextSeq: 0),
           mockQueues: initTable[string, MockQueue](),
           context: AssertionContext(strict: true),
           generation: 0, createdAt: getMonoTime(), active: true)

# Note: `Timeline` is an `object` embedded by value within `ref Verifier`.
# Mutations via `v.timeline.record(...)` work through the ref indirection
# because `v.timeline` is an lvalue (the field access on a ref object
# yields a mutable reference to the embedded object). A regression test
# in `tests/test_timeline.nim` exercises this — passing `v.timeline` to
# a `var Timeline` parameter is legal Nim when `v` is a ref.

proc registerMock*(v: Verifier, pluginName: string, m: Mock) =
  if pluginName notin v.mockQueues:
    v.mockQueues[pluginName] = MockQueue(mocks: initDeque[Mock]())
  v.mockQueues[pluginName].mocks.addLast(m)

proc popMatchingMock*(v: Verifier, pluginName, procName, fingerprint: string): Option[Mock] =
  if pluginName notin v.mockQueues: return none(Mock)
  let q = addr v.mockQueues[pluginName]
  if q.mocks.len == 0: return none(Mock)
  let head = q.mocks[0]
  if head.procName == procName and head.argFingerprint == fingerprint:
    return some(q.mocks.popFirst())
  # inAnyOrder: try all positions. Capture the match BEFORE mutating
  # the deque so the returned Mock is the one we intended to remove.
  if v.context.inAnyOrderActive:
    for i in 0 ..< q.mocks.len:
      if q.mocks[i].procName == procName and q.mocks[i].argFingerprint == fingerprint:
        let matched = q.mocks[i]
        var tmp: seq[Mock]
        for j in 0 ..< q.mocks.len:
          if j != i: tmp.add(q.mocks[j])
        q.mocks.clear()
        for m in tmp: q.mocks.addLast(m)
        return some(matched)
  return none(Mock)
```

### 4.2 Thread-local verifier stack (sandbox.nim)

```nim
# nimfoot/core/sandbox.nim
var verifierStack* {.threadvar.}: seq[Verifier]

proc pushVerifier*(v: Verifier): Verifier =
  ## Push `v` onto the current thread's stack. Returns `v` for chaining.
  verifierStack.add(v)
  v

proc popVerifier*(): Verifier =
  ## Pop and return the top verifier. Increments its generation counter.
  ## Raises AssertionDefect if the stack is empty (internal invariant violation).
  doAssert verifierStack.len > 0, "popVerifier called on empty stack"
  result = verifierStack.pop()
  inc(result.generation)
  result.active = false

proc currentVerifier*(): Verifier {.inline.} =
  ## Top-of-stack accessor. Returns nil if the stack is empty.
  if verifierStack.len == 0: nil else: verifierStack[^1]

template sandbox*(body: untyped) =
  ## Low-level lexical scope. See `test:` (§9) for the integration-
  ## flavored variant that adds pending-ops check and unittest hooks.
  let v = pushVerifier(newVerifier())
  try:
    body
  finally:
    discard popVerifier()
    v.verifyAll()
```

### 4.3 verifyAll() algorithm

```nim
proc verifyAll*(v: Verifier) =
  ## Check the three guarantees. Raises the FIRST violation.
  ## No `{.raises.}` annotation: Defects are non-enumerable in Nim
  ## idiom, and `newException` can raise `OutOfMemDefect`.

  # Guarantee (2): every recorded interaction explicitly asserted.
  var unasserted: seq[Interaction] = @[]
  for i in v.timeline.unasserted: unasserted.add(i)
  if unasserted.len > 0:
    raise newUnassertedInteractionsDefect(v, unasserted)

  # Guarantee (3): every registered mock consumed.
  var unusedMocks: seq[Mock] = @[]
  for pluginName, q in v.mockQueues:
    for m in q.mocks: unusedMocks.add(m)
  if unusedMocks.len > 0:
    raise newUnusedMocksDefect(v, unusedMocks)

  # Guarantee (1) is enforced eagerly at TRM fire time in the plugin body
  # (via popMatchingMock returning none + raising UnmockedInteractionDefect).
```

### 4.4 State transitions

```
    ┌─────────────┐  newVerifier()
    │  non-existent│──────────────┐
    └─────────────┘              │
                                 ▼
                          ┌──────────────┐
                          │  constructed │  active=true, gen=0
                          └──────┬───────┘
                                 │ pushVerifier()
                                 ▼
                      ┌─────────────────────┐
                  ┌──▶│  on verifierStack  │◀──┐
                  │   └──────────┬──────────┘    │ (nested sandbox:)
   TRM fires      │              │              │ pushVerifier(v')
  record/match    │              │              │
                  └──────────────┤              │
                                 │ body ends    │
                                 ▼              │
                          ┌──────────────┐      │
                          │    popped    │──────┘
                          └──────┬───────┘ active=false, gen+=1
                                 │ verifyAll()
              ┌──────────────────┼──────────────────┐
              │                  │                  │
              ▼                  ▼                  ▼
        (no violation)    (unasserted)        (unused mocks)
         returns           raises             raises
                           Unasserted         UnusedMocks
                           Defect             Defect
```

### 4.5 Nesting, exceptions, and failure cases

**Nested sandboxes.** `sandbox:` is lexically nestable; each nested block pushes its own verifier. TRMs always address the top of stack. An inner verifier is `verifyAll`'d before the outer finishes — so inner guarantee violations surface first. Plugin registration is **idempotent** (§5); plugin-install-count is refcounted so inner blocks don't uninstall plugins the outer relies on.

**Exception inside sandbox body.** `sandbox:` uses `try: body; finally: popVerifier(); v.verifyAll()`. If the user body raises, the verifier is still popped and verified. If `verifyAll` then also raises, the user exception is shadowed; document this tradeoff as "when both a user exception and a guarantee violation occur, the guarantee violation wins." (Bigfoot does the same — forces visibility of the more informative failure.) Mitigation: when we see both, we attach the user exception's message as `parent` on the Defect so it's recoverable from the exception chain.

**The three failure cases:**

1. **Unmocked call inside sandbox** — TRM body runs `popMatchingMock` → `none` → raises `UnmockedInteractionDefect` at the call site. Stack unwinds to the `sandbox:`/`test:` finally, which still runs `verifyAll` (may surface additional violations). Error message includes the unmocked call's fingerprint and the nearest-matching expected mocks for hints.
2. **Unasserted interaction** — `verifyAll` detects `unasserted` entries in timeline and raises `UnassertedInteractionsDefect` with a bulleted list of every recorded-but-unasserted interaction (plugin-formatted).
3. **Unused mock** — `verifyAll` detects non-empty `mockQueues` and raises `UnusedMocksDefect` with a bulleted list of every registered-but-unconsumed mock (with file:line from registration site).

## 5. Plugin contract

### 5.1 Base types and registration

```nim
# nimfoot/plugin.nim
type
  Plugin* = ref object of RootObj
    name*: string
    enabled*: bool

  MockResponse* = ref object of RootObj
    ## Base type. Subclass per plugin with concrete response fields.

# ---- Registration ----------------------------------------------------
var pluginRegistry {.global.}: seq[Plugin] = @[]

proc registerPlugin*(p: Plugin) =
  ## Idempotent: a second register of the same name replaces.
  for i, existing in pluginRegistry:
    if existing.name == p.name:
      pluginRegistry[i] = p
      return
  pluginRegistry.add(p)

proc enabledPlugins*(): seq[Plugin] =
  for p in pluginRegistry:
    if p.enabled: result.add(p)

proc pluginByName*(name: string): Plugin =
  for p in pluginRegistry:
    if p.name == name: return p
  nil

# ---- Base methods ----------------------------------------------------
method assertableFields*(p: Plugin, i: Interaction): seq[string] {.base.} =
  ## Subclasses enumerate the fields that the TEST must explicitly
  ## assert. Defaults to an empty list (no mandated assertions — which
  ## is a plugin bug if the plugin records any interactions).
  @[]

method formatInteraction*(p: Plugin, i: Interaction): string {.base.} =
  ## Bigfoot-style multi-line rendering:
  ##   <plugin>.<procName> at <file>:<line>:<col>
  ##     args:
  ##       <key>: <value>
  ##     response:
  ##       <key>: <value>
  result = p.name & "." & i.procName & " at " & i.site.file &
    ":" & $i.site.line & ":" & $i.site.column & "\n"
  if i.args.len > 0:
    result.add "  args:\n"
    for k, v in i.args: result.add "    " & k & ": " & v & "\n"
  # response rendering is plugin-specific via a dispatch on i.response.

method matches*(p: Plugin, procName, fingerprint: string): bool {.base.} = false

method formatError*(p: Plugin, i: Interaction): string {.base.} =
  ## Longer-form rendering used in error-message bodies. Defaults to
  ## `formatInteraction`; plugins override for context-rich hints.
  p.formatInteraction(i)
```

### 5.2 Plugin state machine

```
  registered  ──── registerPlugin(p) called at module init ────▶
      │
      │  p.enabled := config.enabledPlugins.contains(p.name)
      ▼
    active  ◀────────┐ (multiple sandboxes may activate in sequence)
      │              │
      │ sandbox: { … │
      ▼              │
  recording  (TRMs fire; each records an Interaction on currentVerifier())
      │              │
      │ } body ends  │
      ▼              │
   verified  ── verifier.verifyAll() called; mockQueues drained ─┘
```

Plugins are **stateless singletons** in the registry. The per-test state lives entirely on the `Verifier` (mock queues and timeline). This means activation counting is not needed at the plugin level — a verifier binds to the stack, all plugins automatically "use" it via the shared `currentVerifier()` accessor.

### 5.3 Shared TRM-body helpers

Every plugin TRM body has the same skeleton. Three separate primitives would be easy to mis-order (e.g. calling `recordAndReturn` before `nimfootGuard`, or wrapping them in a block so `{.dirty.}` name leakage breaks). We therefore expose a **single combinator** that plugin authors call once per TRM body. The three-template form below (Count / Guard / InterceptBody) is the underlying primitive; plugin authors should use `nimfootInterceptBody` exclusively.

```nim
# nimfoot/plugin.nim — shared TRM helpers

template nimfootGuard*(plugin: Plugin, procName: string): untyped {.dirty.} =
  ## Defense 6 primitive. Checks that a verifier is live AND active.
  ## Exposed for advanced plugin authors; most plugins use
  ## `nimfootInterceptBody` which calls this internally.
  let _nfVerifier = currentVerifier()
  if _nfVerifier.isNil:
    raise newLeakedInteractionDefect(getThreadId(), instantiationInfo())
  if not _nfVerifier.active:
    raise newPostTestInteractionDefect(_nfVerifier.name, _nfVerifier.generation,
                                       plugin.name, procName)

template nimfootInterceptBody*(plugin: Plugin, procName: string,
                               fingerprint: string,
                               responseType: typedesc,
                               spyBody: untyped): untyped {.dirty.} =
  ## Canonical TRM body combinator. Plugin authors place this as the
  ## sole body of their TRM (after the rewrite-count guard emitted by
  ## the macro scaffolding). It handles, in order:
  ##   1. nimfootCountRewrite()   — Defense 3 cap check
  ##   2. nil / active verifier guard — Defense 6
  ##   3. popMatchingMock — Guarantee (1) lookup
  ##   4. timeline recording — so Guarantee (2) can see it
  ##   5. spy passthrough OR UnmockedInteractionDefect
  ##
  ## `responseType` is the plugin's concrete MockResponse subclass
  ## (e.g. `HttpMockResponse`). The plugin's subclass MUST override
  ## `method realize(r: responseType): ReturnT` to convert a registered
  ## mock into the value the intercepted proc should return.
  ##
  ## `spyBody` is the raw-call expression to evaluate when passthrough
  ## fires (wrapped in `{.noRewrite.}:` so it doesn't re-trigger the TRM).
  nimfootCountRewrite()
  let _nfVerifier = currentVerifier()
  if _nfVerifier.isNil:
    raise newLeakedInteractionDefect(getThreadId(), instantiationInfo())
  if not _nfVerifier.active:
    raise newPostTestInteractionDefect(_nfVerifier.name, _nfVerifier.generation,
                                       plugin.name, procName)
  let _nfMockOpt = _nfVerifier.popMatchingMock(plugin.name, procName, fingerprint)
  # Record BEFORE returning so timeline mirrors call order even when the
  # body raises later (e.g. an unmocked sibling in the same test).
  discard _nfVerifier.timeline.record(plugin, procName,
      fingerprintArgs(procName, fingerprint), nil, instantiationInfo())
  if _nfMockOpt.isNone:
    if plugin.supportsPassthrough() and plugin.passthroughFor(procName):
      {.noRewrite.}:
        spyBody
    else:
      raise newUnmockedInteractionDefect(plugin.name, procName, fingerprint,
                                         instantiationInfo())
  else:
    responseType(_nfMockOpt.get.response).realize()
```

Every plugin's `MockResponse` subclass overrides `realize` to convert the registered payload into the intercepted proc's return value:

```nim
method realize*(r: MockResponse): auto {.base.} =
  ## Base: plugin authors MUST override. Default is a loud failure so
  ## a plugin that forgets the override is caught at the first call.
  raise newException(NimfootDefect,
    "MockResponse.realize must be overridden by each plugin's subclass")
```

Plugin authors typically write:

```nim
method realize*(r: HttpMockResponse): Response =
  Response(version: "1.1", status: $r.status, headers: r.headers,
           body: r.body, bodyStream: newStringStream(r.body))
```

**Why one combinator rather than three primitives?** The three-template sequence (CountRewrite + Guard + RecordAndReturn) is structurally required; mis-ordering or wrapping in a nested block silently breaks `{.dirty.}` name leakage. Enforcing the sequence via a single call eliminates that class of plugin-author mistake. The underlying primitives remain exported for advanced authors who want finer-grained composition (e.g. a plugin that customizes its error-message rendering and wants to read `_nfVerifier` between guard and popMatchingMock).

#### 5.3.1 Canonical TRM body skeleton

A complete, copy-paste-ready TRM body for a non-async intercept:

```nim
template mockTRM_computeThing*{computeThing(a0, a1)}(a0, a1: int): int =
  nimfootInterceptBody(
    mockPluginInstance,                     # Rule 6: pluginInstance is module-global
    "computeThing",                         # Rule 10: exact proc-name string
    fingerprintOf("computeThing", @[$a0, $a1]),  # Rule 3: fingerprint from rendered args
    MockUserResponse[int],                   # Rule 12: concrete MockResponse subclass
    spyBody = {.noRewrite.}: computeThing(a0, a1))  # Rule 13: passthrough body
```

Annotation (Plugin Authoring Rule coverage):
- `nimfootInterceptBody` call satisfies Rules 3 (Defense 3 cap), 6 (guard), 7 (timeline record), 8 (queue pop), 9 (unmocked raise).
- Explicit `MockUserResponse[int]` resolves I3's compile-time/runtime tension (the concrete `T` is known at TRM emission time).
- `{.noRewrite.}: computeThing(a0, a1)` satisfies Rule 13 (spy passthrough avoids recursion); validated by spike #5 Q1b.

### 5.4 Mock queue structure (FIFO deque)

`std/deques.Deque[Mock]` — O(1) `addLast` and `popFirst`. Keyed per plugin name on the Verifier (`mockQueues: Table[string, MockQueue]`). One queue per plugin means `expect` for httpclient and `expect` for osproc don't interleave; ordering only matters within a plugin's own expectations.

**`inAnyOrder:` relaxation.** When the context flag is set, `popMatchingMock` does a linear scan of the queue for any position match, not just head. This is O(n) per match but `inAnyOrder` is used for small (≤10) interaction blocks in practice (matches bigfoot's usage).

### 5.5 Timeline assertion-queue structure

The timeline is an append-only `seq[Interaction]`. Each Interaction has an `asserted` bool flag. `assert:` DSL calls find the matching interaction (by plugin + procName + fingerprint, FIFO among unasserted) and flip the flag. `verifyAll` then scans for any `asserted == false` entries.

### 5.6 Fire-through layering

By default (spike #7), when a plugin's TRM body calls a **lower-level** instrumented proc, the lower-level proc's TRM also fires. Example: if an httpclient `get` TRM internally calls `request` (which also has a TRM), both TRMs fire and both record interactions.

Plugin authors have three choices per TRM body:

1. **Default (fire-through):** do nothing; both TRMs record. The test must assert both. Use when both layers are semantically meaningful (user cares about both "I called get" and "the request used these headers").
2. **Suppress inner:** wrap the inner call in `{.noRewrite.}:`. Only the outer TRM records. Use when the outer level is the user-facing abstraction and the inner is an implementation detail.
3. **Suppress outer:** intercept only at the inner level; don't register a TRM for the outer. The outer call is opaque to nimfoot. Use when the outer is a thin wrapper.

**Plugin guideline:** document the chosen layering depth for each intercept target. For v0, the httpclient plugin chooses option 3 (`request` is the intercept point; `get/post/put/...` compose through it without their own TRMs). See §12.2 for rationale and tradeoff.

### 5.7 Spy mode

`passthrough = true` on a mock registration activates spy mode. In spy mode, the TRM body:

1. Still records an Interaction on the timeline (so `assert:` block works)
2. Calls the REAL proc wrapped in `{.noRewrite.}:` to avoid recursion (spike #5 Q1b validated this)
3. Uses the real return value as the interaction's `response`

**Plugin opt-in.** Spy mode is off by default. Plugins opt in by overriding two base methods:

```nim
# Example: the MockPlugin supports passthrough for every user proc,
# because the user's own code is always safe to call.
method supportsPassthrough*(p: MockPlugin): bool = true
method passthroughFor*(p: MockPlugin, procName: string): bool = true

# Example: the osproc plugin cannot passthrough startProcess safely
# (would spawn a real subprocess) but CAN passthrough waitForExit
# against a fake Process whose mockId is known.
method supportsPassthrough*(p: OsprocPlugin): bool = true
method passthroughFor*(p: OsprocPlugin, procName: string): bool =
  procName in ["waitForExit", "close", "running"]
```

`nimfootInterceptBody` consults both methods before firing passthrough: `plugin.supportsPassthrough() and plugin.passthroughFor(procName)`. A plugin that leaves `supportsPassthrough` as the base `false` disables spy mode for all its intercepts, regardless of per-proc overrides.

**Return-value wrapping.** When passthrough fires, `spyBody` evaluates the raw call and its value becomes the TRM's return. The plugin's `realize` method is NOT invoked on passthrough — the raw call already produces the correct return type. (`realize` is only called when a registered mock was found.) For plugins whose MockResponse carries metadata the spy wants to record, the plugin author should record that metadata inside the TRM body **around** the `nimfootInterceptBody` call, using the underlying primitives.

Spike #5 Summary: Only `{.noRewrite.}:` **as a statement-block pragma** works. The `{.push noRewrite.}/{.pop.}` form is silently broken in Nim 2.2.6. The capture-via-proc-variable fallbacks (Q2a/Q3a) work for concrete non-generic procs but fail for generic / varargs / default-arg procs.

Standardize on `{.noRewrite.}:`. Fallbacks noted in plugin-authoring.md.

### 5.9 TRM registration order semantics

TRM pattern match is **first-fit** in registration order within a plugin module. Spike #7 verified this: when two TRMs share the same pattern shape but differ in parameter types, the first-registered wins for matching types; subsequent ones are tried only when the first fails to type-check.

Concrete example:

```nim
# plugins/osproc.nim — registered in this order, ONE module scope
template execProcessSeqTRM*{execProcess(c, w, a, e, o)}(...): string = ...
template execProcessArrayTRM0*{execProcess(c, w, a, e, o)}(...): string = ...
# ... arrays 1..8 ...
# Fallback trap MUST be last so it's only tried after every concrete
# container variant has been tested:
template execProcessOpenArrayFallback*{execProcess(c, w, a, e, o)}(...): string = ...
```

If the fallback trap were registered first, it would match every call regardless of container type and the concrete variants would never fire. This is the core rationale for the §11.5 "fallback trap MUST be last" rule.

## 6. `auto.nim` and activation

### 6.1 Activation chain

```
consumer's config.nims  ──  --import:"nimfoot/auto"  ──┐
                            --define:"nimfootActive"   │
                                                       ▼
                                  src/nimfoot/auto.nim (injected into every compilation unit)
                                                       │
                          when defined(nimfootActive): │
                                                       ▼
                                  imports all plugin modules:
                                    import nimfoot/plugins/mock
                                    import nimfoot/plugins/httpclient
                                    import nimfoot/plugins/osproc
                                                       │
                                                       ▼
                                  each plugin module emits its TRMs at module scope
                                  (module-top-level TRMs are the pattern that lets
                                   injection work — spike #1 validated 3+ levels deep)
```

### 6.2 `auto.nim` shape

```nim
## nimfoot/auto.nim
##
## Umbrella TRM module. Imported via `--import:"nimfoot/auto"` so that
## every compilation unit in the consumer's project carries the TRMs
## for all enabled plugins. Gated by `nimfootActive` so production
## builds skip all TRM work entirely.

when defined(nimfootActive):
  # Load core first (plugins reference it).
  import ./plugin
  # Load plugin modules. Each plugin registers itself at module init
  # and emits its TRMs at module scope. Order matters only for
  # deterministic error messages; runtime behavior is independent.
  import ./plugins/mock
  import ./plugins/httpclient
  import ./plugins/osproc

  # Optional FFI audit (Defense 2).
  when defined(nimfootAuditFFI):
    import ./audit_ffi
else:
  # no-op: production build. Zero codegen, zero TRM, zero overhead.
  discard
```

Deliberate choices:
- **`when defined(nimfootActive)` gate** ensures that consumers who forget the `-d:nimfootActive` in their PROD config.nims get zero overhead — the file compiles to nothing.
- **The facade's Defense-1 check** (§3.5, §11.1) ensures consumers who `import nimfoot` but forget to set `-d:nimfootActive` get a loud `{.error.}`. This is orthogonal to auto.nim's no-op path: auto.nim is only reached if `--import:nimfoot/auto` is in the consumer's nim flags.
- **Plugin load order** does not matter for correctness. We load them in a fixed order for deterministic error-message rendering.

### 6.3 Interaction with `nimfootAllowInactive`

Advanced consumer case: a library wants to depend on `nimfoot` types (e.g., re-export `Verifier` for other test helpers to use) but doesn't want to force TRM injection on its downstreams. They compile with `-d:nimfootAllowInactive`. This:
- Suppresses the facade's Defense-1 compile-time error
- Does NOT affect auto.nim — if `-d:nimfootActive` is also unset, TRMs aren't emitted
- Effectively makes nimfoot a "types and API only" dep

Document this in `docs/quickstart.md` as a footnote; it's not commonly needed.

### 7.1 Full type tree

```nim
# nimfoot/core/errors.nim

type

  UnmockedInteractionDefect* = object of NimfootDefect

    procName*: string
    fingerprint*: string
    args*: OrderedTable[string, string]

    nearestMockHints*: seq[string]

  UnassertedInteractionsDefect* = object of NimfootDefect

    ## LeakedInteractionDefect, which fires when the stack is empty.

  PendingAsyncDefect* = object of NimfootDefect
    ## test: template teardown found hasPendingOperations() == true.
    ## Points at the asyncCheck-leak pattern; see Defense 6. No exact
    ## pending count is stored because std/asyncdispatch does not
    ## expose one (see §10.3). When chronos is linked in, the chronos
    ## pending count is appended to the message string.
    testName*: string

  UnmockableContainerDefect* = object of NimfootDefect
    ## An openArray TRM fallback-trap fired: container is not in the
    ## plugin's supported set. See Defense 5.
    procName*: string
    containerType*: string
    site*: tuple[file: string, line, column: int]

  NimfootError* = object of CatchableError
    ## Root of nimfoot's catchable-error tree. Reserved for
    ## programmer-misuse signals that the test harness should be
    ## allowed to catch and continue (e.g. a test intentionally
    ## exercising the DSL to confirm it rejects invalid usage).
    ## Distinct from NimfootDefect so that `except CatchableError`
    ## does not swallow verification failures.

    ## User called `assert CALL:` inside the sandbox body instead of
    ## after it. Assertions must happen at teardown (lexically after
```


### 7.2 When-each-fires table

| Defect | Raised by | Trigger condition |
|---|---|---|
| `UnmockedInteractionDefect` | plugin TRM body (via `nimfootInterceptBody`) | `popMatchingMock` returns `none` and plugin doesn't have passthrough for this proc |
| `UnassertedInteractionsDefect` | `Verifier.verifyAll()` | `timeline.unasserted` has entries |

| `AssertionInsideSandboxError` | `assert CALL:` DSL macro | called inside sandbox body before teardown (detected via `context.assertionsOpen` flag) |

### 8.1 `nimfoot.toml` schema

### 8.2 Discovery algorithm

```nim
# nimfoot/config.nim

proc discoverConfigPath(): Option[string] =
  # 1. Explicit override: NIMFOOT_CONFIG env var wins.
  let envPath = getEnv("NIMFOOT_CONFIG", "")
  if envPath.len > 0:
    if fileExists(envPath): return some(envPath)
    # loud — if user set the env var, they meant it
    raise newException(ValueError,
      "NIMFOOT_CONFIG set to '" & envPath & "' but file does not exist")

  # 2. Walk up from cwd until we hit the first .nimble or nimfoot.toml.
  #    Stop at the filesystem root.
  var dir = getCurrentDir().absolutePath()
  while true:
    let tomlPath = dir / "nimfoot.toml"
    if fileExists(tomlPath): return some(tomlPath)
    # Short-circuit on a .nimble — that marks the project root.
    # If there's no nimfoot.toml at the nimble root, use defaults.
    # walkPattern short-circuits on first match, so we don't scan the
    # whole directory when a .nimble exists.
    for _ in walkPattern(dir / "*.nimble"):
      return none(string)  # project root found without config
    let parent = dir.parentDir()
    if parent == dir: break   # hit filesystem root
    dir = parent
  return none(string)

proc loadConfig(path: Option[string]): NimfootConfig =
  if path.isNone:
    return defaultConfig()
  let toml = parseFile(path.get)
  result = NimfootConfig(
    enabledPlugins: toml["nimfoot"]["enabled_plugins"].getElems.mapIt(it.getStr),
    pluginOptions: initTable[string, TomlValueRef](),
    allowPendingAsync: toml.getOrDefault("nimfoot.allow_pending_async", false).getBool,
    # `sources` records the layering trail for diagnostics. The
    # built-in defaults are always consulted first, then any discovered
    # file overlays. When an error message needs to say "plugin X
    # disabled per [source]", it reads from this list.
    sources: @["builtin-defaults", path.get])
  for pluginName in result.enabledPlugins:
    if pluginName in toml:
      result.pluginOptions[pluginName] = toml[pluginName]
  # Firewall parsed but ignored in v0
  if "firewall" in toml:
    result.firewall = parseFirewallConfig(toml["firewall"])

var configMemo {.threadvar.}: Option[NimfootConfig]

proc getConfig*(): NimfootConfig =
  if configMemo.isNone:
    configMemo = some(loadConfig(discoverConfigPath()))
  configMemo.get
```

### 8.3 Per-test override

A test file can override a config setting via a local pragma or explicit call:

```nim
# Per-file opt-in to pending-async warnings-only
nimfoot.allowPendingAsync()

test "long-running integration":
  ...
```

`nimfoot.allowPendingAsync()` is a thread-local flag that sets the same bit as `[nimfoot].allow_pending_async = true` for the current test process. Scope: file-level (not per-test). If a test needs per-test scope, wrap with `sandbox:` and set/restore manually.

### 8.4 Firewall keys for v0.2 (reserved)

Parsed and stored but ignored at runtime in v0. Documented so consumers can start writing configs that will be honored when v0.2 ships:

- `mode`: `off` (default), `allow_list`, `deny_all`
- `allowed_domains`: list of hostnames allowed even in deny_all mode
- `allowed_processes`: executables allowed to spawn even in deny_all mode

### 8.5 When nimfoot reads the config

`getConfig()` is called **lazily** on first `newVerifier()` invocation (via the first `test:` or `sandbox:` block). Memoization is thread-local (via `{.threadvar.}`). This means:
- Config changes between tests within the same process are not picked up
- Tests can't race on config reads
- No cost for compile-time TRM work

Documentation notes that users who need to swap config mid-process must call `nimfoot.reloadConfig()` explicitly (advanced; exposed but not in runnableExamples).

## 9. `std/unittest` + `unittest2` integration

### 9.1 Module shape

```nim
# nimfoot/integrations/unittest.nim
##
## Integration with std/unittest and unittest2. Provides the `test:`
## template that wraps the standard test/suite templates with:
##   1. per-test verifier push/pop
##   2. pending-ops check (Defense 6)
##   3. verifyAll on exit (Defense 4)
##   4. quit hook for best-effort diagnostic flush

import std/macros
import std/exitprocs
import ../core/verifier
import ../core/sandbox
import ../core/errors

# Compile-time switch between std/unittest and unittest2. Users opt in
# to unittest2 explicitly via `-d:nimfootUnittest2`; nimfoot does NOT
# auto-detect via a `compiles` probe because the probe can fail for
# reasons unrelated to unittest2 presence (transitive import cycles,
# nimble path ordering) and produces confusing backend-selection bugs.
when defined(nimfootUnittest2):
  import unittest2 as backend
  const backendName = "unittest2"
else:
  import std/unittest as backend
  const backendName = "std/unittest"

export backend
```

`-d:nimfootUnittest2` is the single authoritative switch. Documentation in §9.6 tells users to add the define to their `config.nims` when they want the unittest2 backend.

### 9.2 `test:` template expansion

```nim
var exitHookRegistered {.threadvar.}: bool = false

template test*(name: string, body: untyped) =
  bind pushVerifier, popVerifier, hasPendingOperations
  bind newVerifier, verifyAll
  bind newPendingAsyncDefect

  # Register the exit hook once per thread (Defense 4).
  # Nim 2.0+ uses std/exitprocs.addExitProc; the older system.addQuitProc
  # is deprecated.
  if not exitHookRegistered:
    addExitProc(verifyAllOnExit)
    exitHookRegistered = true

  backend.test name:
    let _nfV = pushVerifier(newVerifier(name))
    try:
      body
      # Defense 6: pending-ops check BEFORE pop, so the next test
      # doesn't inherit leaked coroutines.
      if hasPendingOperations() and not getConfig().allowPendingAsync:
        raise newPendingAsyncDefect(name)
    finally:
      discard popVerifier()
      _nfV.verifyAll()   # raises if Guarantee 2 or 3 violated

template suite*(name: string, body: untyped) =
  backend.suite name:
    body   # no per-suite verifier; tests handle their own
```

The try/finally guarantees `verifyAll` runs even when the body raises. A raise from `verifyAll` shadows any body-raised exception; we treat that as intentional (more-informative-failure-wins rule from §4.5).

### 9.3 Async teardown sequence

```
enter test "name"
  │
  ├── pushVerifier(newVerifier("name"))
  │
  ├── try:
  │     body                        # user test code; TRMs fire
  │     if hasPendingOperations():
  │       raise PendingAsyncDefect  # Defense 6
  │
  └── finally:
        popVerifier()
        verifier.verifyAll()         # Defense 4 + guarantee checks
exit test
```

### 9.4 Quit-hook flush

```nim
proc verifyAllOnExit() {.noconv.} =
  ## Registered via std/exitprocs.addExitProc. Best-effort: the
  ## process is going down, so we write diagnostics to stderr.
  while verifierStack.len > 0:
    let v = popVerifier()
    try:
      v.verifyAll()
    except NimfootDefect as e:
      stderr.writeLine "nimfoot: unverified verifier '" & v.name &
        "' at exit — " & e.msg
```

This catches the `os.quit()` escape path from Defense 4. It cannot fail the process (we're already quitting), but the user sees the violation instead of a clean green run.

### 9.5 Failure surfacing

- `check` and `require` come from the unittest/unittest2 backend and work unchanged inside `test:`. They raise `UnitTestError` / `require`-specific errors that `std/unittest` catches internally. nimfoot's `Defect` errors escape backends' catch because unittest only catches `CatchableError`.
- When nimfoot raises a `NimfootDefect`, the test fails with the defect's message. The backend's output formatter reports the test as failed with the nimfoot message shown.
- `fail("reason")` from the backend works and is unaffected.

### 9.6 Compile-time backend switch — concrete code

```nim
when defined(nimfootUnittest2):
  import unittest2 as backend
else:
  import std/unittest as backend
```

**Usage.** To run the suite against unittest2, compile with `-d:nimfootUnittest2`. Without the define, nimfoot uses `std/unittest`. There is no auto-detection: an explicit define is mandatory because `when compiles(...)` probes on module imports are unreliable in Nim 2.2.6 (transitive import cycles and nimble path ordering can cause false negatives that masquerade as "unittest2 not installed"). See `tests/test_integration_unittest2.nim` for the live build flag and CI matrix in §13.5.

### 9.7 addExitProc registration

`std/exitprocs.addExitProc` is called **once per thread**. The `exitHookRegistered` threadvar guards the registration. Re-registration is a no-op (Nim allows it but we save the stack frame). The older `system.addQuitProc` is deprecated in Nim 2.0+ and is not used.

## 10. Async model

### 10.1 Why the chronos `complete` template fails under TRM (spike #6 Q4)

A TRM body that synthesizes a response for an `{.async.}` proc must return a `Future[T]`. The obvious inline construction:

```nim
var f = newFuture[string]("mocked")
f.complete("response body")
return f
```

fails under chronos because `complete` is a template that calls `getSrcLocation()`, which in turn expects the caller's lexical position. When `complete` is expanded inside a TRM body being pattern-matched, the lexical position isn't stable, and the chronos macro emits an unresolvable reference.

**Fix:** delegate construction to a runtime proc declared outside the TRM:

```nim
# nimfoot/futures.nim (asyncdispatch branch)
proc makeCompletedFuture*[T](value: sink T, label: string = ""): Future[T] =
  result = newFuture[T](label)
  result.complete(value)

# chronos branch (behind `when defined(chronos)`)
when defined(chronos):
  import chronos
  proc makeCompletedFutureChronos*[T](value: sink T, label: string = ""): Future[T] =
    result = newFuture[T](label)
    result.complete(value)
```

Inside the plugin TRM:

```nim
# Canonical async TRM shape. See §5.3.1 and §11.5.1 for the general
# pattern; only the spyBody and responseType differ per plugin.
template httpclientGetAsyncTRM*{
    request(c, url, httpMethod, body, headers, multipart)}(
    c: AsyncHttpClient, url: string, httpMethod: HttpMethod = HttpGet,
    body: string = "", headers: HttpHeaders = nil,
    multipart: MultipartData = nil): Future[AsyncResponse] =
  nimfootInterceptBody(
    httpclientPluginInstance,
    "request",
    fingerprintHttpRequest(url, httpMethod, body, headers, multipart),
    HttpAsyncMockResponse,
    spyBody = {.noRewrite.}:
      request(c, url, httpMethod, body, headers, multipart))
```

`HttpAsyncMockResponse.realize()` returns a `Future[AsyncResponse]` synthesized via `makeCompletedFuture` (§10.2). The `makeCompletedFuture` call is a plain proc call; the TRM pattern-matcher does not look inside proc bodies, so chronos's `complete` template runs with stable lexical context.

### 10.2 `when defined(chronos)` branching

`futures.nim` always exports the `asyncdispatch` helpers (stdlib is always available). chronos helpers only compile when chronos is importable:

```nim
# nimfoot/futures.nim
import std/asyncdispatch
export asyncdispatch

proc makeCompletedFuture*[T](value: sink T, label: string = ""): Future[T] =
  result = newFuture[T](label)
  result.complete(value)

proc makeFailedFuture*[T](err: ref Exception, label: string = ""): Future[T] =
  result = newFuture[T](label)
  result.fail(err)

when defined(chronos):
  import chronos
  export chronos

  # chronos Futures are a different type (chronos.Future[T]).
  # Plugins that target chronos-API procs use these variants.
  proc makeCompletedFutureChronos*[T](value: sink T, label: string = ""): chronos.Future[T] =
    result = chronos.newFuture[T](label)
    chronos.complete(result, value)

  proc makeFailedFutureChronos*[T](err: ref Exception,
      label: string = ""): chronos.Future[T] =
    result = chronos.newFuture[T](label)
    chronos.fail(result, err)
```

Plugins detect at macro expansion time which flavor of `Future[T]` their intercepted proc returns and dispatch to the appropriate helper. For v0, stdlib httpclient uses `asyncdispatch.Future`; consumers using chronos's own http client (out of scope for v0 — users would write their own plugin or wait for v0.2) would use the chronos helpers.

### 10.3 `asyncdispatch` branching for `hasPendingOperations()`

```nim
# nimfoot/core/sandbox.nim (selected)
import std/asyncdispatch

proc hasPendingOperations*(): bool =
  result = asyncdispatch.hasPendingOperations()
  when defined(chronos):
    import chronos
    result = result or chronos.pendingFutureCount() > 0
```

If a project uses chronos, `chronos.pendingFutureCount()` augments the stdlib check.

**On pending count:** `std/asyncdispatch` as of Nim 2.2.6 exposes only the boolean `hasPendingOperations()` — there is no public `numPendingFutures`, and the dispatcher's internal tables are private. nimfoot therefore reports **presence only** in the `PendingAsyncDefect` message, not an exact count. When chronos is linked in, the chronos side can contribute an exact count via `pendingFutureCount()`; the stdlib side contributes only "pending: yes". Appendix B lists exposing an upstream `asyncdispatch.numPendingFutures` accessor as a v0.1 candidate.

### 10.4 Template stubs for plugin TRM bodies

Plugin authors write TRMs that produce `Future[T]` using these stubs:

```nim
# Used inside the plugins/httpclient.nim async TRMs
template asyncMockResponse*(mock: Mock, T: typedesc): Future[T] =
  makeCompletedFuture(T(mock.response.HttpMockResponse), "httpclient")

template asyncMockFailure*(err: ref Exception, T: typedesc): Future[T] =
  makeFailedFuture[T](err, "httpclient-failure")
```

### 10.5 `PendingAsyncDefect` trigger

```nim
# Inside the `test:` template (Defense 6)
if hasPendingOperations() and not getConfig().allowPendingAsync:
  raise newPendingAsyncDefect(name)
```

This fires **before** `popVerifier` so the test that leaked is named in the error. Spike #8 validated this catches cross-test contamination at the guilty boundary.

### 10.6 Tradeoff: why no generation-counter detection

Spike #8 Q4 showed that a generation counter on the verifier does NOT detect asyncCheck-leaked TRM firings against the NEXT test's verifier (the TRM resolves `currentVerifier()` at fire time, which returns the live next-test verifier — indistinguishable from a legitimate fire). The pending-ops teardown check catches this at the guilty test's boundary **before** contamination spreads.

We accept the residual risk that a user who disables the pending-ops check (`-d:nimfootAllowPendingAsync`) loses this protection. Document as a known limitation.

### 11.1 Defense 1: import guard

### 11.2 Defense 2: FFI scope

**Part 1: Error-message preamble**

Every nimfoot Defect's top-level message includes a footer line:

```
(nimfoot intercepts Nim source calls only. FFI ({.importc.}, {.dynlib.}, {.header.})
is not intercepted in v0. See docs/concepts.md#scope.)
```

Emitted by the base `NimfootDefect` constructor:

```nim
const FFIScopeFooter* = "\n(nimfoot intercepts Nim source calls only. " &
  "FFI ({.importc.}, {.dynlib.}, {.header.}) is not intercepted in v0. " &
  "See docs/concepts.md#scope.)"

proc newNimfootDefect*[T: NimfootDefect](msg: string): ref T =
  result = newException(T, msg & FFIScopeFooter)
```

**Part 2: README preamble**

Top of `README.md` carries a `!! SCOPE` call-out block (same text as above) so users know before they install what is and isn't covered.

**Part 3: `-d:nimfootAuditFFI` macro sketch**

```nim
# nimfoot/audit_ffi.nim — only compiled under -d:nimfootAuditFFI
##
## Compile-time audit of FFI pragmas in the transitive dep tree.
## Emits a hint per intercepted-potentially-escape callsite so the
## user has a list of "unknown risks" to manually review.

import std/[macros, strutils, sets]

var auditReport {.compileTime.} = newSeq[string]()

macro auditModule*(m: typed) =
  ## Walk `m`'s AST looking for {.importc.}, {.dynlib.}, {.header.}
  ## pragmas on proc declarations. Emit a {.hint.} for each.
  result = newStmtList()
  for node in m.recursiveWalk:   # pseudocode; actual walker in implementation
    if node.kind == nnkProcDef:
      for pragma in node.pragma:
        if pragma.kind == nnkExprColonExpr and
           pragma[0].strVal.toLowerAscii in ["importc", "dynlib", "header"]:
          let line = $node.lineInfoObj
          auditReport.add line & ": FFI call " & node.name.strVal
  if auditReport.len > 0:
    # Emit at the end of compilation
    hint("nimfoot FFI audit: " & $auditReport.len & " unaudited FFI callsites:\n" &
         auditReport.join("\n"))
```

The `recursiveWalk` is pseudocode; the full implementation uses `std/macros.walkPackedType` or a hand-rolled AST walker to traverse every typed symbol in the current compilation unit. Consumer opts in with `-d:nimfootAuditFFI` and gets a list of every intercepted module's FFI-carrying proc. Converts an unknowable risk into a knowable one.

### 11.3 Defense 3: cap tripwire

**File:** `src/nimfoot/core/compile_guards.nim`

```nim
## Defense 3 — per-hloBody TRM cap tripwire.
##
## Nim silently stops rewriting after ~19 TRMs in one hloBody (module
## top-level OR proc body). This cap is undocumented and not signaled
## at runtime. We use a compile-time counter to error out BEFORE we
## hit the silent-cap threshold.
##
## Usage: every plugin TRM body MUST call nimfootCountRewrite() as
## its FIRST statement. The counter is reset per hloBody by Nim's
## own macro-expansion semantics (each proc body is a fresh hloBody).

import std/macros

const NimfootCapThreshold = 15  # safety margin below Nim's ~19 cap

var nfRewriteCount {.compileTime.} = 0   # compile-time counter

template nimfootCountRewrite*(): untyped =
  static:
    inc(nfRewriteCount)
    when nfRewriteCount > NimfootCapThreshold:
      {.error: "nimfoot: more than " & $NimfootCapThreshold &
        " TRM rewrites in a single hloBody. Nim's rewrite engine " &
        "silently drops rewrites past ~19. Split the test with " &
        "`nimfoot.sandbox:` sub-blocks (each sub-block runs in " &
        "its own proc, getting its own rewrite budget), or refactor " &
        "this proc into smaller helpers.".}
```

**Semantics:** the counter tracks **total TRM emissions per compilation unit**, not per-hloBody. Nim does not expose a per-hloBody counter to user macros, and an attempted `static: nfRewriteCount = 0` reset at `test:` entry races nondeterministically against TRM-body increments during compiler elaboration (elaboration order for nested `static:` + TRM bodies is not guaranteed by the Nim 2.2 compiler). We therefore do NOT reset per-test in v0.

**Threshold rationale:** 15 is a heuristic for typical test modules of fewer than ~40 rewrites (each hloBody inside a compilation unit tops out at ~19 per spike #2, and tests normally invoke 1–3 distinct TRMs per hloBody). If a user's module exceeds the threshold, the remediation is to split the test file into multiple `test_*.nim` files — each compilation unit gets its own counter. The threshold is deliberately conservative; a false positive produces a loud compile-time error with a clear message, which is strictly better than a silent drop.

**Deferred:** precise per-hloBody accounting requires a `{.compileTime.}` `Table[string, int]` keyed by file-plus-test-name and a macro pass that resets on each test-proc body entry. This is tractable but adds compile-time Table overhead. Deferred to v0.2 pending user feedback on whether the simple counter misfires in practice. See Appendix B.

### 11.5 Defense 5: openArray container coverage + fallback trap

**Supported container set per plugin:**

1. `seq[T]` — always
2. `array[N, T]` for any compile-time `N` — via generic TRM variants
3. `string` — when `T = char`
4. Bare `openArray[T]` literal at the call site — via a fallback trap

**Fallback trap TRM shape:**

```nim
# Example from plugins/osproc.nim for execProcess with args: openArray[string]
template execProcessOpenArrayFallback*{
    execProcess(cmd, workingDir, args, env, options)}(
    cmd: string, workingDir: string, args: openArray[string],
    env: StringTableRef, options: set[ProcessOption]): string =
  # This TRM only fires if NONE of the concrete-container TRMs match.
  # The container type is unsupported; report loudly.
  nimfootCountRewrite()
  raise newUnmockableContainerDefect(
    procName = "execProcess",
    containerType = $type(args),
    site = instantiationInfo())
```

This TRM **must be the last-registered** among the execProcess variants. TRM pattern match is first-fit in registration order (verified by spike #7 for layering), so `seq[string]` and `array[N, string]` variants match first; the bare `openArray[string]` shape only matches if the user passed an exotic container that elaborates to `openArray` without matching a concrete container.

**Container variants emitted by `osproc.nim` (canonical shape):**

```nim
# seq[string] variant — written by hand, one per spawn proc.
template execProcessSeqTRM*{
    execProcess(cmd, workingDir, args, env, options)}(
    cmd: string, workingDir: string, args: seq[string],
    env: StringTableRef, options: set[ProcessOption]): string =
  nimfootInterceptBody(
    osprocPluginInstance,
    "execProcess",
    fingerprintExecProcess(cmd, workingDir, args, env, options),
    OsprocExecProcessResponse,
    spyBody = {.noRewrite.}: execProcess(cmd, workingDir, args, env, options))
```

Array variants for `N in 0..8` are emitted via a macro. Because ``quote do`` cannot interpolate a backtick into an identifier, the macro must build the template name via `newIdentNode`:

#### 11.5.1 Worked example: emitting per-N array TRMs

```nim
import std/macros

macro emitExecProcessArrayVariants(maxN: static[int]): untyped =
  result = newStmtList()
  for n in 0 .. maxN:
    let tmplName = newIdentNode("execProcessArrayTRM" & $n)
    # Manual AST construction: nnkTemplateDef with a TRM pattern arm.
    # Pattern-side names (c, w, a, e, o) are bound-name references
    # that MUST match the parameter-list names below. If they diverge,
    # the TRM silently fails to match (validated by spike #3).
    let tdef = nnkTemplateDef.newTree(
      nnkPostfix.newTree(ident"*", tmplName),
      newEmptyNode(),               # term-rewriting pattern generics
      newEmptyNode(),               # term-rewriting pattern default
      nnkFormalParams.newTree(
        ident"string",              # return type
        newIdentDefs(ident"c", ident"string"),
        newIdentDefs(ident"w", ident"string"),
        newIdentDefs(ident"a",
          nnkBracketExpr.newTree(ident"array", newLit(n), ident"string")),
        newIdentDefs(ident"e", ident"StringTableRef"),
        newIdentDefs(ident"o",
          nnkBracketExpr.newTree(ident"set", ident"ProcessOption"))),
      nnkPragma.newTree(),
      newEmptyNode(),
      # TRM pattern: `{execProcess(c, w, a, e, o)}` becomes
      # an nnkStmtList child where the first node carries the curly
      # pattern. The actual emission uses nnkCurly in 2.2.6; the
      # canonical accessor is `newCall("execProcess", ...)` wrapped in
      # nnkPattern. See spike #3 sub-experiment 3.2 for the exact AST.
      wrapInTRMPattern(
        newCall("execProcess", ident"c", ident"w", ident"a", ident"e", ident"o"),
        body = newCall("nimfootInterceptBody",
          ident"osprocPluginInstance",
          newLit("execProcess"),
          newCall("fingerprintExecProcess",
            ident"c", ident"w", ident"a", ident"e", ident"o"),
          ident"OsprocExecProcessResponse",
          nnkExprEqExpr.newTree(ident"spyBody",
            nnkPragmaBlock.newTree(
              nnkPragma.newTree(ident"noRewrite"),
              newCall("execProcess",
                ident"c", ident"w", ident"a", ident"e", ident"o"))))))
    result.add(tdef)

emitExecProcessArrayVariants(8)
```

The `wrapInTRMPattern` helper encapsulates the Nim 2.2.6 TRM AST shape (per spike #3). Plugin authors who want to emit TRMs programmatically should use this helper rather than hand-building the term-rewriting pattern node. Sibling macros in `plugins/httpclient.nim` and `plugins/mock.nim` use the same helper — see those modules for the canonical invocations. All other per-variant TRM emissions in this document follow the same pattern; we spell out only this one example in full.

**Pattern-variable vs argument-name relationship.** Inside `{exec(c, w, a, e, o)}`, the identifiers `c, w, a, e, o` are **TRM pattern variables** — each must correspond 1:1 to a parameter declared in the following formal-params list. If the pattern declares `{exec(cmd, ...)}` but the params list names the first arg `c`, the pattern never matches because the bound name is `cmd` with no matching param. Spike #3 sub-experiment 3.4 documents this failure mode.

### 11.6 Defense 6: async and threads

**Nil-verifier and inactive-verifier checks are performed by `nimfootInterceptBody`** (§5.3). The underlying primitive:

```nim
template nimfootGuard*(plugin: Plugin, procName: string): untyped {.dirty.} =
  let _nfVerifier = currentVerifier()
  if _nfVerifier.isNil:
    raise newLeakedInteractionDefect(getThreadId(), instantiationInfo())
  if not _nfVerifier.active:
    raise newPostTestInteractionDefect(_nfVerifier.name, _nfVerifier.generation,
                                       plugin.name, procName)
```

The active-flag check wires `PostTestInteractionDefect` into a real trigger path: when a TRM fires against a verifier that has already been popped (e.g. a late-completing Future from an `asyncCheck` leak that escaped the pending-ops check), the plugin and generation counter are captured for diagnostics.

On a worker thread, the `LeakedInteractionDefect` hits Nim's unhandled-exception hook and terminates the process with a stack trace (verified by spike #9 Q3). On an async-leaked fire, the exception propagates through the dispatcher's poll loop (spike #8 Q3 verified it escapes `poll()` with an async traceback).

**Pending-ops check at teardown** — shown in §9.2. Fires `PendingAsyncDefect` at the guilty test boundary, before the next test starts, avoiding spike #8 Q4's cross-test contamination.

**Documentation:**

`docs/quickstart.md` has a "What's not supported" section covering:

- **`--threads:on` user-code threading** — unsupported in v0. TRMs on worker threads raise `LeakedInteractionDefect` and crash the process. Rationale: inheritance requires log-merging semantics, gcsafe verifier transport, and per-threading-library wrappers (createThread, spawn, malebolgia, taskpools, weave). Deferred to v0.2. See understanding doc §Defense 6 and spike #9 for feasibility prototype.
- **`asyncCheck` inside tests** — unsupported in v0. Pending-ops check fires `PendingAsyncDefect` at test exit. Users must `waitFor` / `await` all futures they launch inside a test body.
- **Escape hatch** — `-d:nimfootAllowPendingAsync` for tests with legitimate long-running async (e.g., connection-pool warmup). Treats the pending-ops check as informational.

## 12. Plugin-specific designs

### 12.1 MockPlugin (`src/nimfoot/plugins/mock.nim`)

A **generic** plugin for mocking arbitrary user-declared procs. The user declares a mockable proc in their code, then registers expectations for it at test time. No stdlib-specific TRMs; the TRMs are generated per-user-proc by an `expect:` / `assert:` DSL macro.

**User-facing flow:**

```nim
# user's source.nim
proc computeThing(x, y: int): int =
  httpGet("https://api.example.com/calc?x=" & $x & "&y=" & $y).parseInt

# user's test.nim
import nimfoot, nimfoot/plugins/mock

sandbox:
  expect computeThing(2, 3):
    respond value: 42
  let result = someHigherFunction(2, 3)
  check result == 43
  assert computeThing(2, 3):
    responded value: 42
```

**DSL keyword choice: `value:` not `return:`.** `return` is a Nim reserved word that the parser always elaborates as `nnkReturnStmt`, so writing `respond return: 42` would not parse as the `nnkExprColonExpr` shape a DSL expects. We therefore use `value:` throughout — valid `nnkExprColonExpr(ident"value", <expr>)`.

**Macro expansion plan:**

The `expect` macro inspects the call AST `computeThing(2, 3)`, captures the proc name and arg fingerprint, and:
1. Calls `MockPlugin.registerMock(computeThing_fingerprint, response)` on the current verifier
2. At first invocation, emits a TRM for that proc (guarded by a `{.compileTime.}` registry so we don't emit duplicates)

**DSL macro skeleton:**

```nim
# nimfoot/plugins/mock.nim

type
  MockPlugin* = ref object of Plugin
  MockUserResponse*[T] = ref object of MockResponse
    ## Generic in the intercepted proc's return type. The concrete `T`
    ## is known at macro-emission time because the `expect` macro
    ## resolves the proc symbol and reads its return type via
    ## `macros.getTypeImpl`. The runtime `returnValue` is the actual
    ## value the user passed to `respond value: <expr>`.
    returnValue*: T

method realize*[T](r: MockUserResponse[T]): T = r.returnValue

let mockPluginInstance* = MockPlugin(name: "mock", enabled: true)
registerPlugin(mockPluginInstance)

method assertableFields*(p: MockPlugin, i: Interaction): seq[string] =
  @["return"]

macro expect*(call: untyped, body: untyped): untyped =
  ## `expect computeThing(2, 3): respond value: 42`
  ## generates:
  ##   1. a TRM for computeThing (if not already emitted)
  ##   2. a registerMock call on the current verifier
  ##
  ## The concrete T of `MockUserResponse[T]` is inferred from the
  ## resolved proc symbol's return type via macros.getTypeImpl.
  expectKind(call, nnkCall)
  let procName = call[0]
  let args = call[1..^1]
  let retType = procReturnType(call)   # helper: getTypeImpl on resolved sym
  # Find `respond value: VALUE` in body
  var valueExpr: NimNode
  for stmt in body:
    if stmt.kind == nnkCommand and stmt[0] == ident"respond":
      for kv in stmt[1..^1]:
        if kv.kind == nnkExprColonExpr and kv[0] == ident"value":
          valueExpr = kv[1]
  # Emit TRM if first time for this proc (tracked in compileTime registry)
  let trm = emitMockTRMIfNeeded(procName, args, retType)
  let register = quote do:
    registerMock(currentVerifier(), "mock",
      newMock($`procName`, argsFingerprint(`call`),
        MockUserResponse[`retType`](returnValue: `valueExpr`),
        instantiationInfo()))
  result = newStmtList(trm, register)
```

**TRM emission per user proc:**

For each user proc mentioned in an `expect`, the macro emits (on first reference) a TRM matching that proc's exact signature. Since we don't know the signature at `expect`-parse time, we use the Nim compiler's symbol table: `macros.getTypeImpl` on the resolved proc symbol gives us the signature, and we construct a TRM from it:

Each emitted TRM is constructed via manual `nnkTemplateDef` AST (rather than `quote do:`) because the TRM pattern arg list and argument binding require precise control over symbol identity — see §11.5.1 for the worked example. The emitted shape:

```nim
# Generated for `computeThing(x, y: int): int`:
template mockTRM_computeThing*{computeThing(a0, a1)}(a0, a1: int): int =
  nimfootInterceptBody(
    mockPluginInstance,
    "computeThing",
    fingerprintOf("computeThing", @[$a0, $a1]),
    MockUserResponse[int],
    spyBody = {.noRewrite.}: computeThing(a0, a1))
```

The concrete `MockUserResponse[int]` (not a bare `MockUserResponse`) is resolved at macro-emission time from the proc's return type. `nimfootInterceptBody` then reads the typed response at runtime via `MockUserResponse[int](resp).realize()`.

**Generic / varargs / default-arg procs:** the macro delegates to the user-author to specify the signature explicitly via `nimfoot.mockable(computeThing, (x, y: int): int)` or similar when the generic-heuristic fails. For v0, we support the common case of non-generic fixed-arity procs; document generic/varargs as "requires manual TRM" and link to the plugin-authoring guide.

### 12.2 httpclient plugin (`src/nimfoot/plugins/httpclient.nim`)

**Intercept targets** (from spike #3 and understanding doc §5):

The `std/httpclient` plugin intercepts the **base `request` proc** rather than each convenience wrapper. Reason (spike #7): `get`, `post`, `put`, `delete`, `patch`, `head`, `getContent`, `postContent`, `deleteContent`, `putContent`, `patchContent` all internally call `request`. When a TRM fires for `request`, the convenience wrappers' calls are captured through the fire-through layering. Tradeoff:
- Pro: one TRM to maintain per sync/async variant; consistent assertion shape across wrappers
- Con: `assert get(client, url): ...` has to match the `request` interaction, not a `get` interaction — we work around this by having the assert DSL pattern-match on `procName == "request" and args[\"method\"] == \"GET\"`

**Full TRM enumeration** (what v0 ships):

| # | TRM | Sync/Async | Notes |
|---|---|---|---|
| 1 | `request(HttpClient, url: string, ...)` | sync | base request; captures method, url, body, headers |
| 2 | `request(HttpClient, url: Uri, ...)` | sync | Uri overload |
| 3 | `request(AsyncHttpClient, url: string, ...)` | async | async variant |
| 4 | `request(AsyncHttpClient, url: Uri, ...)` | async | async Uri |
| 5 | `newHttpClient(...)` | ctor | lifecycle; returns a fake HttpClient |
| 6 | `newAsyncHttpClient(...)` | ctor | async lifecycle |
| 7 | `close(HttpClient)` | lifecycle | |
| 8 | `close(AsyncHttpClient)` | lifecycle | |
| 9 | `downloadFile(HttpClient, url, filename)` | sync | downloads go through `request` plus filesystem; we intercept separately to avoid accidental writes |
| 10 | `downloadFile(AsyncHttpClient, url, filename)` | async | |

For the convenience wrappers (`get`, `post`, `put`, `delete`, `patch`, `head`, `getContent`, `postContent`, `deleteContent`, `putContent`, `patchContent`):
- They are NOT given their own TRMs.
- They compose through `request`. The user's `expect get(client, url): ...` DSL rewrites the registration to target `request(client, url, method="GET", ...)`.

**Wrapper canonicalization table.** When the user's DSL references a convenience wrapper, the `expect` macro emits a `request`-shaped mock with the defaults std/httpclient actually passes. These defaults reflect `std/httpclient` as of Nim 2.2.6; if the stdlib changes them, the plugin must be re-audited.

| Wrapper | method | body | headers | multipart | Notes |
|---|---|---|---|---|---|
| `get(c, url)` | `"GET"` | `""` | `nil` | `nil` | |
| `post(c, url, body, multipart)` | `"POST"` | `body` | `nil` | `multipart` | |
| `put(c, url, body, multipart)` | `"PUT"` | `body` | `nil` | `multipart` | |
| `delete(c, url)` | `"DELETE"` | `""` | `nil` | `nil` | |
| `patch(c, url, body)` | `"PATCH"` | `body` | `nil` | `nil` | |
| `head(c, url)` | `"HEAD"` | `""` | `nil` | `nil` | |
| `getContent(c, url)` | `"GET"` | `""` | `nil` | `nil` | returns body string |
| `postContent(c, url, body, multipart)` | `"POST"` | `body` | `nil` | `multipart` | returns body string |
| `deleteContent(c, url)` | `"DELETE"` | `""` | `nil` | `nil` | returns body string |
| `putContent(c, url, body)` | `"PUT"` | `body` | `nil` | `nil` | returns body string |
| `patchContent(c, url, body)` | `"PATCH"` | `body` | `nil` | `nil` | returns body string |
| `downloadFile(c, url, filename)` | `"GET"` | `""` | `nil` | `nil` | plus `writeFile` side effect |

**OPTIONS, TRACE, and other non-wrapper methods.** Users wanting to mock OPTIONS, TRACE, or any non-wrapper HTTP method must call `request(c, url, HttpOptions)` directly and the `expect` DSL accepts a `request(method: HttpOptions)` form to match. nimfoot v0 does not ship dedicated wrappers for these because std/httpclient does not.

**Content-family wrappers.** The `*Content` wrappers (`getContent`, `postContent`, `deleteContent`, `putContent`, `patchContent`) are convenience procs that call the matching verb wrapper and then read `response.body`. They route through `request` indirectly. Mocking the underlying verb (e.g., `expect get(c, url): respond status: 200, body: "ok"`) intercepts both the verb call and any `*Content` wrapper above it; no separate `*Content` mock is needed.

**DSL → mock resolution.** When the user writes `expect get(client, url): respond status: 200, body: "ok"`, the `expect` macro looks up `get` in the canonicalization table above and emits the equivalent `request`-level mock registration:

```nim
registerMock(currentVerifier(), "httpclient",
  newMock("request",
    fingerprintHttpRequest(url, "GET", "", nil, nil),
    HttpMockResponse(status: 200, body: "ok",
                     headers: newHttpHeaders()),
    instantiationInfo()))
```

`assert get(client, url): responded status: 200` resolves the same way: the DSL rewrites the assertion-search predicate to `procName == "request" and args["method"] == "GET" and args["url"] == url`.

**Count:** Base `request` × sync/async × (url: string | url: Uri) = 4 TRMs. Lifecycle (new + close) × sync/async = 4 TRMs. `downloadFile` × sync/async = 2 TRMs. **Total ~10 TRMs**, not 30.

The "30 TRMs" figure from the understanding doc includes the convenience wrappers as separate entries; the design choice to intercept at `request` reduces this to ~10. The DSL handles the user-facing mapping.

**Note on `Uri | string` typeclass union.** `request` accepts `url: Uri | string` via an overload (two separate procs, not a typeclass union). Each overload is its own TRM. The DSL normalizes: if the user writes `expect get(client, "http://..."): ...`, the macro looks up the overload table and registers for the `string` variant; if they write `expect get(client, parseUri("http://...")): ...`, it registers for the `Uri` variant. Tradeoff: two registrations per mock if the user wants to accept either form; we doc this as "be consistent with your URL type."

**Default values must match stdlib exactly.** `httpclient.request` in Nim 2.2.6 declares `headers: HttpHeaders = nil` (NOT `newHttpHeaders()`) and `sslContext = getDefaultSSL()`. TRM signatures reproduce these expressions **exactly** (per Plugin Authoring Rule 1):

```nim
template requestSyncTRM*{
    request(c, url, httpMethod, body, headers, multipart)}(
    c: HttpClient, url: string, httpMethod: HttpMethod = HttpGet,
    body: string = "", headers: HttpHeaders = nil,
    multipart: MultipartData = nil): Response =
  nimfootInterceptBody(
    httpclientPluginInstance,
    "request",
    fingerprintHttpRequest(url, httpMethod, body, headers, multipart),
    HttpMockResponse,
    spyBody = {.noRewrite.}:
      request(c, url, httpMethod, body, headers, multipart))
```

The compiler elaborates `nil` at TRM-definition time to match the actual stdlib default. If the stdlib changes the default, we catch it with our self-tests (flag matrix in §13). See the version-compatibility note below.

**Version-compatibility fragility (Plugin Authoring Rule 1).** If stdlib changes a parameter default (for example, `headers: HttpHeaders = nil` becomes `headers: HttpHeaders = newHttpHeaders()`), the TRM pattern stops binding silently: the compiler elaborates the TRM's default at definition time, the user's call site elaborates the stdlib default, and the two fingerprints no longer match. The httpclient plugin's CI must run against each supported Nim version. v0 targets Nim 2.2.6 exclusively; v0.1 widens to 2.0.x and 2.2.x with version-conditional TRM bodies (`when (NimMajor, NimMinor) == (2, 0): ... else: ...`) if needed.

**getContent/postContent layering decision:** intercept at `request` level. Documented in plugin-authoring.md as the canonical example of "intercept depth choice."

**openArray handling for `httpclient.addFiles`:** the `addFiles` proc takes `files: openArray[(string, string)]`. Per Defense 5, emit:
- `seq[(string, string)]` TRM
- `array[N, (string, string)]` for N in 0..8
- `openArray[(string, string)]` fallback trap

Users passing tuples with named fields (`tuple[name: string, path: string]`) — different type — hit the trap. Doc explains how to add named-tuple variants.

**Fingerprint helper and response subclasses.** TRM bodies delegate the record/pop/raise sequence to `nimfootInterceptBody` (§5.3); the plugin only needs to provide (a) a fingerprint helper that canonicalizes args and (b) a `realize` override per response subclass.

```nim
# nimfoot/plugins/httpclient.nim — plugin-specific helpers

type
  HttpMockResponse* = ref object of MockResponse
    status*: int
    headers*: HttpHeaders
    body*: string

  HttpAsyncMockResponse* = ref object of MockResponse
    status*: int
    headers*: HttpHeaders
    body*: string

proc fingerprintHttpRequest*(url: string, httpMethod: HttpMethod,
                             body: string, headers: HttpHeaders,
                             multipart: MultipartData): string =
  ## Canonicalize the request args for mock lookup.
  $httpMethod & " " & url & " body=" & body & " hdr=" & $headers

method realize*(r: HttpMockResponse): Response =
  Response(
    version: "1.1",
    status: $r.status,
    headers: r.headers,
    body: r.body,
    bodyStream: newStringStream(r.body))

method realize*(r: HttpAsyncMockResponse): Future[AsyncResponse] =
  makeCompletedFuture(
    AsyncResponse(version: "1.1", status: $r.status,
                  headers: r.headers, body: r.body),
    "httpclient.request")
```

Async variants use `makeCompletedFuture` inside their `realize` method; the async TRM body shape is otherwise identical to sync (see §10.1 `httpclientGetAsyncTRM`).

### 12.3 osproc plugin (`src/nimfoot/plugins/osproc.nim`)

**Intercept targets:**

| Category | Procs | Count |
|---|---|---|
| Spawn | `execProcess`, `execCmd`, `execCmdEx`, `startProcess`, `execProcesses` | 5 |
| Control | `waitForExit`, `kill`, `terminate`, `suspend`, `resume`, `close`, `running`, `processID` | 8 |
| Streams | (deferred — users intercept the underlying stream proc) | 0 |

**Per-proc TRM count** varies with `openArray` arity. Taking `execProcess` as the worst case:

```nim
# execProcess full signature:
proc execProcess(command: string, workingDir: string = "",
                 args: openArray[string] = [],
                 env: StringTableRef = nil,
                 options: set[ProcessOption] = {poStdErrToStdOut, poUsePath, poEvalCommand}): string
```

Per Defense 5: `args` has type `openArray[string]`, so we emit:
- 1 TRM for `args: seq[string]`
- 9 TRMs for `args: array[0..8, string]` (via `emitArrayVariants(8)`)
- 1 TRM for `args: string` if applicable (not — string isn't a container of string; N/A here)
- 1 fallback trap TRM for `args: openArray[string]`

**Total for execProcess:** 1 + 9 + 1 = 11 TRMs. Similar counts for each spawn variant that takes openArray. Control procs have no openArray → 1 TRM each.

**`extern: "nosp$1"` mitigation (Plugin Authoring Rule 11):**

`std/osproc` procs all carry `{.extern: "nosp$1".}`. The `$1` expands to the proc name, so `execProcess` becomes C symbol `nospexecProcess`. If our TRM body calls a helper named `execProcess` anywhere, the linker sees two `nospexecProcess` symbols and errors.

**Workaround:** every osproc helper called by a TRM body is prefixed `nimfootRaw`:

```nim
# Bad (link collision risk):
proc execProcessImpl(cmd: string, args: seq[string]): string = ...

# Good:
proc nimfootRawExecProcess(cmd: string, args: seq[string]): string = ...
```

The `nimfootRaw` prefix is reserved for osproc-plugin internal helpers; document in plugin-authoring.md.

**Anonymous tuple returns:**

`execCmdEx` returns `tuple[output: string, exitCode: int]` (named). The TRM signature reproduces the named form:

```nim
template execCmdExTRM*{
    execCmdEx(cmd, options, env, workingDir, input)}(
    cmd: string,
    options: set[ProcessOption] = {poStdErrToStdOut, poUsePath, poEvalCommand},
    env: StringTableRef = nil, workingDir: string = "",
    input: string = ""): tuple[output: string, exitCode: int] =
  nimfootInterceptBody(
    osprocPluginInstance,
    "execCmdEx",
    fingerprintExecCmdEx(cmd, options, env, workingDir, input),
    OsprocExecCmdExResponse,
    spyBody = {.noRewrite.}:
      execCmdEx(cmd, options, env, workingDir, input))
# OsprocExecCmdExResponse.realize returns the named tuple:
# method realize*(r: OsprocExecCmdExResponse): tuple[output: string, exitCode: int] =
#   (output: r.output, exitCode: r.exitCode)
```

A TRM written as `: (string, int)` (positional) would NOT match calls to the real `execCmdEx` (distinct types). Plugin Authoring Rule 10.

**Platform-gated `hasData`:**

`osproc.hasData` is `when defined(windows)` on one branch, POSIX-specific on another. Our TRM matches whatever branch is compiled:

```nim
when defined(windows):
  template hasDataWinTRM*{hasData(p)}(p: Process): bool = ...
else:
  template hasDataPosixTRM*{hasData(p)}(p: Process): bool = ...
```

Both branches generate the same TRM body shape.

**`startProcess` return type `Process`:** we cannot synthesize a real `Process` (it wraps an actual OS handle). Instead, we return a **fake Process** — a sentinel object with a magic PID that our control-proc TRMs recognize and handle without calling stdlib's control procs. The fake Process carries the mock's expected exit code, output, etc. Pattern:

```nim
type
  NimfootFakeProcessTag* = object
    mockId*: int
    expectedExit*: int
    expectedOutput*: string
    expectedStderr*: string

# A private compile-time table maps Process instances (by identity)
# back to their tag. This is the mechanism that lets control-proc
# TRMs recognize a fake Process without mutating std/osproc internals.
var fakeProcessTags {.threadvar.}: Table[int, NimfootFakeProcessTag]

proc makeFakeProcess(mock: OsprocStartProcessResponse): Process =
  # Allocates a Process ref and records its tag in the thread-local
  # table under the ref's internal id. The ref's OS handle fields are
  # left at their defaults; user code never reads them because every
  # control proc is intercepted.
  result = Process()
  fakeProcessTags[cast[int](result)] = NimfootFakeProcessTag(
    mockId: mock.mockId, expectedExit: mock.exitCode,
    expectedOutput: mock.output, expectedStderr: mock.stderr)

proc fakeTagOf(p: Process): Option[NimfootFakeProcessTag] =
  let key = cast[int](p)
  if key in fakeProcessTags: some(fakeProcessTags[key]) else: none(NimfootFakeProcessTag)
```

Example control-proc TRM that reads the tag:

```nim
template waitForExitTRM*{waitForExit(p, timeout)}(
    p: Process, timeout: int = -1): int =
  nimfootInterceptBody(
    osprocPluginInstance,
    "waitForExit",
    fingerprintOf("waitForExit", @[$cast[int](p), $timeout]),
    OsprocWaitForExitResponse,
    spyBody = {.noRewrite.}: waitForExit(p, timeout))

# The plugin's realize method for WaitForExitResponse reads the tag
# the user's startProcess mock installed and returns its expectedExit:
method realize*(r: OsprocWaitForExitResponse): int =
  # When the test wrote `expect startProcess(...): respond exitCode: 42`,
  # the fake Process's tag carries expectedExit: 42. The
  # waitForExit mock can default to echoing that tag, or the user can
  # override explicitly via a separate waitForExit expect.
  if r.useTagExit:
    # pull from the currently-being-intercepted Process arg
    fakeTagOf(r.process).get.expectedExit
  else:
    r.exitCode
```

This is an implementation detail; users see a `Process` that behaves per their mock.

## 13. Compilation model, testing strategy, and ergonomics

### 13.1 Consumer activation — concrete `config.nims`

```nim
# In the consumer's test/config.nims or nim.cfg:
--import:"nimfoot/auto"
--define:"nimfootActive"
--warning:UnusedImport:off        # recommended; nimfoot injects imports users never reference
--trmacros:on                      # explicit (default is on for --trmacros, was --patterns)

# Optional audit
# --define:"nimfootAuditFFI"
```

For production builds, the consumer's regular `config.nims` OMITS these lines. Nothing changes in production — `auto.nim` becomes a no-op, facade's Defense-1 guard can be suppressed with `-d:nimfootAllowInactive`.

### 13.2 `--trmacros:off` for production

Note: `--patterns:off` is the DEPRECATED alias for `--trmacros:off`. Consumer docs use `--trmacros:off`. When off, ALL TRMs are disabled (not just nimfoot's), so use this only when you explicitly want a TRM-free build. Most consumers rely on the `-d:nimfootActive` gate instead, which affects only nimfoot.

### 13.3 Production build with `nimfootActive` undefined

- `auto.nim`'s TRMs never emit (the `when defined(nimfootActive):` is false).
- The facade is safe to import if `-d:nimfootAllowInactive` is set; otherwise, a clean prod build skips `import nimfoot` entirely.
- **Zero codegen difference vs not importing nimfoot:** proven by self-test that diffs the compiled output of a small program with and without nimfoot in the import graph.

### 13.4 Self-test structure

```
tests/
├── all_tests.nim            # imports every test_*.nim; nim c -r tests/all_tests
├── test_verifier.nim        # Verifier type unit tests
├── test_timeline.nim        # Timeline + Interaction ops
├── test_sandbox.nim         # push/pop/currentVerifier + nested sandbox:
├── test_mock_plugin.nim     # MockPlugin DSL + TRM emission (implement first)

│                            #   D1: compile-fail without flags (runs nim c --dry, checks stderr)
```

### 13.5 Flag matrix

`nimfoot.nimble` defines `task test` with the matrix:

```nim
task test, "Run the full test matrix":
  # refc + sync
  exec "nim c --gc:refc --define:nimfootActive --import:nimfoot/auto -r tests/all_tests.nim"
  # orc + sync
  exec "nim c --gc:orc --define:nimfootActive --import:nimfoot/auto -r tests/all_tests.nim"
  # refc + unittest2
  exec "nim c --gc:refc --define:nimfootActive --define:nimfootUnittest2 --import:nimfoot/auto -r tests/all_tests.nim"
  # orc + unittest2
  exec "nim c --gc:orc --define:nimfootActive --define:nimfootUnittest2 --import:nimfoot/auto -r tests/all_tests.nim"
  # orc + chronos (chronos requires orc/arc; refc is NOT a supported
  # chronos GC so we don't run a refc+chronos cell — chronos upstream
  # docs explicitly flag refc as unsupported for --gc:refc since its
  # 4.x release. If chronos gains refc support upstream, add the cell.)
  exec "nim c --gc:orc --define:nimfootActive --define:chronos --import:nimfoot/auto -r tests/all_tests.nim"

task test_defenses, "Run the Defense-1 compile-fail check":
  # Separate task because D1 expects a non-zero exit code
  let r = execShellCmd "nim c --dry-run tests/defense1_probe.nim 2>&1"
  doAssert r != 0, "Defense 1 did not fire as expected"

task test_fast, "Run one config for quick iteration":
  exec "nim c --gc:orc --define:nimfootActive --import:nimfoot/auto -r tests/all_tests.nim"
```

Default `nimble test` runs the full matrix. CI runs `task test`.
